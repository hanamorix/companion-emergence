# v0.0.7-alpha Comprehensive Audit

**Date:** 2026-05-11
**Scope:** Full-tree audit per `docs/superpowers/specs/2026-05-11-v0.0.7-comprehensive-audit-design.md`.
**Status:** Resolved. Phase A/B found 12 issues, Phase D smoke added F-201, Phase E triaged every finding as `[fix]`, and Phase F landed all fixes on `main`. Phase G release prep for v0.0.7-alpha is in progress.

## Summary

**12 findings total.** One P1 (caught by Hana's Phase D smoke), 6× P2, 5× P3 — no P0. Axis breakdown: 6× code, 2× feature, 4× hygiene, 1× polish, 3× quality (some findings count under two axes; numbers don't sum to 12). Layer breakdown: 7× brain (one of which is brain+ts), 1× ts, 4× infra-docs.

The tree is in genuinely good shape after the v0.0.4 → v0.0.6 batch of feature work. The Rust shell is clean across every axis (cargo clippy, target-cfg correctness, command surface, persona validation parity, CSP). The TS frontend has 32 passing tests, deterministic, and respects reduced-motion + focus-visible globally. The Python brain has a small cluster of SQLite-hygiene + validation-drift findings that mostly trace back to "this seam was added after the canonical pattern was established and missed the pattern." Docs lag the release cadence — the most user-visible drift is the status banner on README/INSTALL still claiming v0.0.1-alpha.

The headless smoke (Phase B) validated both supervisor sticky-session contracts end-to-end (`session_snapshot` on sweep + buffer survives; `session_finalized` on 24h + buffer deleted). The shutdown-drain path was NOT covered by the smoke — that's where Hana's Phase D smoke caught **F-201**, the headline finding: Cmd-Q + reopen doesn't resume the conversation because the shutdown drain is destructive (still using v0.0.5-era `close_stale_sessions` instead of the v0.0.6 non-destructive `snapshot_stale_sessions`) AND the renderer always creates a fresh session on mount with no reattach logic. The supervisor learned the new pattern; the app-quit path didn't.

One smoke-driven brain finding (F-011) about persistently-failing extraction wasting LLM calls without backoff.

- **P1** — affects user-visible feature. Default fix.
- **P2** — maintainer pain, latent risk. Default defer.

### P1

#### F-201 — Cmd-Q + reopen does NOT resume the conversation (Phase B contract broken on shutdown path)

- **Axis:** feature
- **Layer:** brain + ts
- **Location:** `brain/bridge/server.py:299-326` (`_drain_sessions_blocking` calls destructive `close_stale_sessions`); `app/src/components/ChatPanel.tsx:259-263` (renderer always creates fresh session on mount).
- **Observation:** Phase D smoke (Hana, 2026-05-11) caught this directly: after a long streaming reply, Cmd-Q + reopen the app, and the conversation is gone — Nell does not pick up mid-thread. Two compounding bugs:
  1. **Shutdown drain is destructive.** `server.py::_drain_sessions_blocking` calls `close_stale_sessions(silence_minutes=0)`, which runs `close_session` per live session — extracting memories AND deleting buffer + cursor + registry entry. The buffer file is gone after Cmd-Q. (Phase B introduced `snapshot_stale_sessions` for the non-destructive 5-min sweep path; the shutdown drain was not updated.)
  2. **Renderer always creates a fresh session on mount.** `ChatPanel.tsx:259-263` reads `sessionRef.current` (which starts as null on mount), and if null calls `newSession(persona)` — generating a new session_id. There's no attempt to look up "what was the last live session for this persona." There's no `/sessions/active` endpoint in the bridge either.
- **Why it matters:** The headline feature of v0.0.6-alpha (sticky sessions) is silently broken for the most common "I'm done for now" gesture — Cmd-Q. Users who close + reopen the app get an amnesic Nell, exactly the bug Phase B was meant to fix. The brief-absence case (≤24h silence with the app still running) still works because the supervisor's sweep is non-destructive; only the app-quit code path is broken.
- **Suggested fix:** Three-part change, ordered by smallest payoff first:
  1. **Replace `close_stale_sessions` with `snapshot_stale_sessions` in `_drain_sessions_blocking`** (`server.py:309, 320`). Memories still extract on shutdown; buffer + cursor stay on disk. The in-memory `_SESSIONS` registry is gone after process exit anyway — buffer survives.
  2. **Add a `GET /sessions/active` endpoint** that returns the most-recent-turn session_id for a persona by scanning `<persona>/active_conversations/*.jsonl` ts fields. Filter to sessions whose last turn is < `finalize_after_hours` hours old (i.e. not finalize-eligible).
  3. **Renderer attach-on-mount logic.** In `ChatPanel.tsx`, before calling `newSession(persona)`, call `/sessions/active` for the persona. If it returns a session_id, set `sessionRef.current` to that. Only fall back to `newSession` if no active session exists.
- **Effort:** medium. Each part is small (10-30 lines); the integration test is the bigger piece — needs a "boot bridge, send turns, kill bridge ungracefully, restart bridge, attach, verify history" scenario. Worth doing right.
- **Triage:** [fix]
- **Note:** Memories extracted during the destructive drain ARE still durable in `MemoryStore`, so the ambient recall block in `build_system_message` would still surface "yesterday's conversation" as durable memories on reopen. But the raw-transcript-replay path that Phase B added is the *fidelity* path — and that's what Hana noticed missing.

### P2

#### F-001 — EmbeddingCache missing WAL + busy_timeout pragmas

- **Axis:** code
- **Layer:** brain
- **Location:** `brain/memory/embeddings.py:83`
- **Observation:** `EmbeddingCache.__init__` opens `sqlite3.connect(str(db_path))` without configuring `journal_mode=WAL` or `busy_timeout`. `MemoryStore`, `HebbianMatrix`, and `SoulStore` all configure both. The supervisor opens EmbeddingCache per-tick from a background thread; if asyncio chat handlers happen to be reading the same DB through any future code path, contention would surface as `OperationalError: database is locked` with no retry window.
- **Why it matters:** Inconsistent locking behaviour across stores; the next bug that depends on concurrent embedding access will be platform-specific and hard to repro.
- **Suggested fix:** Mirror the `MemoryStore` pattern — `PRAGMA journal_mode=WAL` (after a guarded `PRAGMA integrity_check` if you want) and `PRAGMA busy_timeout=5000`.
- **Effort:** trivial
- **Triage:** [fix]

#### F-002 — WorksStore missing busy_timeout pragma

- **Axis:** code
- **Layer:** brain
- **Location:** `brain/works/store.py:83-89` (`_connect`)
- **Observation:** `WorksStore._connect` sets `journal_mode=WAL` and `foreign_keys=ON` but does NOT set `busy_timeout`. The fresh-connection-per-call pattern means each call could hit a writer lock and immediately fail with `database is locked`.
- **Why it matters:** Same concurrency rationale as F-001. `save_work` already wraps inserts in `BEGIN IMMEDIATE`, which makes the lock window wider — exactly when busy_timeout earns its keep.
- **Suggested fix:** Add `conn.execute("PRAGMA busy_timeout = 5000")` to `_connect`, alongside the existing pragmas.
- **Effort:** trivial
- **Triage:** [fix]

#### F-003 — persona-name validation drift between paths.py and setup.py

- **Axis:** code
- **Layer:** brain
- **Location:** `brain/paths.py:41-58` (`get_persona_dir`) vs `brain/setup.py:53-65` (`validate_persona_name`)
- **Observation:** `validate_persona_name` enforces `^[A-Za-z0-9_-]{1,40}$` (strict regex). `get_persona_dir` only blocks `/ \ { } . ..`, so a programmatic caller bypassing `validate_persona_name` could land at `get_persona_dir("Nell Smith")` or `get_persona_dir("nell@home.com")` and create an unexpected directory. Both validators raise `ValueError` so a single shared helper is straightforward.
- **Why it matters:** Validation drift between two siblings is a classic security/consistency latent — `validate_persona_name` is the canonical contract per CLAUDE.md (rule referenced in prior audit's P2-1).
- **Suggested fix:** Make `get_persona_dir` call `validate_persona_name` from `brain.setup` (or move the regex to `brain.paths` and have `setup` re-export it). Pick whichever doesn't introduce a circular import.
- **Effort:** small
- **Triage:** [fix]

#### F-004 — claude_tool_searcher stub accepted by PersonaConfig

- **Axis:** feature
- **Layer:** brain
- **Location:** `brain/search/claude_tool_searcher.py` + `brain/persona_config.py` (acceptlist)
- **Observation:** `claude_tool_searcher` raises `NotImplementedError` on every method. `PersonaConfig` accepts `"claude-tool"` as a valid searcher value and the CLI surfaces it. A user who selects it will get a runtime crash on first heartbeat that tries to dispatch a search. This was P2-4 in the 2026-05-07 audit; still open in v0.0.6.
- **Why it matters:** Documented option that crashes — exactly the documented-but-stubbed pattern the spec calls out under feature axis.
- **Suggested fix:** Either implement `claude-tool` properly OR remove it from `PersonaConfig`'s allowlist + drop the CLI flag value + delete the stub file. The latter is faster and matches user-surface principle (don't expose what isn't real).
- **Effort:** small (removal); medium (implementation)
- **Triage:** [fix]

#### F-007 — Staged-but-unsent image URL leaks on unmount

- **Axis:** polish
- **Layer:** ts
- **Location:** `app/src/components/ChatPanel.tsx:205`, 156-160 (cleanup), 279 (track-on-send)
- **Observation:** `URL.createObjectURL(file)` runs at line 205 for the staged image preview. The URL is added to `trackedUrlsRef` ONLY when the message is sent (line 279). If the user stages an image and then unmounts ChatPanel (persona switch, app quit, route change) WITHOUT sending it, the URL is held only in React state — which gets thrown away on unmount — and never revoked. Net effect: one leaked blob URL per unsent staged image per session.
- **Why it matters:** Memory + handle leak. For a single image it's trivial; for users who stage-then-cancel repeatedly across a long session it accumulates. Browsers eventually reclaim object URLs but the contract is explicit revocation.
- **Suggested fix:** Add the previewUrl to `trackedUrlsRef` at line 205 right after creation. Remove-and-revoke is still safe — `Set.has()` makes the unmount sweep idempotent. Alternatively, useEffect cleanup keyed on `stagedImage?.previewUrl` that revokes when the staged image is replaced.
- **Effort:** trivial
- **Triage:** [fix]

#### F-011 — Persistently-failing extraction wastes LLM calls every sweep

- **Axis:** quality
- **Layer:** brain
- **Location:** `brain/ingest/pipeline.py::extract_session_snapshot` + the `snapshot_stale_sessions` sweep loop
- **Observation:** Smoke run with FakeProvider (which returns "DREAM: …" text, unparseable as JSON) showed every supervisor sweep tick re-runs extraction on the same buffer, fails on the same content, advances no cursor, and tries again 5 minutes later. With `tick_interval=60s` and `silence_minutes=5`, that's up to 12 wasted extraction calls per hour for any session whose content the model can't parse. The retry-on-cursor-unchanged design is correct (we don't want to skip turns) but there's no per-session failure budget.
- **Why it matters:** Cost amplification + log noise. A single weird session can balloon API spend; under cost-pressure cadences (long sessions, high-traffic days), a wedged extractor is a real cost line.
- **Suggested fix:** Add a per-session failure counter to the cursor sidecar (or a sibling file). After N consecutive `conversation_snapshot_failed` for the same cursor, advance the cursor anyway (giving up on those turns — they stay in the buffer but won't be re-extracted) OR pause the session in extraction for an exponential backoff window. Backoff is the cheaper option. N=3 + 10-min backoff is a reasonable starting point.
- **Effort:** small
- **Triage:** [fix]

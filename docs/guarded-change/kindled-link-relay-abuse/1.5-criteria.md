# 1.5 — Acceptance criteria

All automated (pytest against `relay.dev_relay.create_app` via Starlette `TestClient`
— the existing `test_dev_relay.py` harness — plus a unit on `_payload_text`).

## Envelope size cap

- **C-1 (gating).** A `/envelope` POST whose serialised body exceeds
  `RELAY_MAX_ENVELOPE_BYTES` → HTTP 413, and the envelope is NOT stored
  (`store.fetch` for that mailbox unchanged).
- **C-2 (gating).** A normal-size envelope still pushes → 200 + stored (regression of
  the existing happy path).

## Per-mailbox queue depth

- **C-3 (gating).** After `RELAY_MAX_QUEUE_DEPTH` envelopes are queued to one mailbox,
  the next push → HTTP 429 (or 507) and is NOT stored; the queue length stays at the
  cap. Existing queued envelopes remain fetchable.
- **C-4 (gating).** `ack`-ing envelopes frees capacity → a push succeeds again after
  the queue drops below the cap.

## Keyspace bound — global mailbox cap (NOT registration-gating)

**Stage-3 blocker resolution:** `/envelope` push stays OPEN to any mailbox (store-and-
forward MUST accept deposits to recipient mailboxes that are not yet registered — the
recipient self-registers lazily on its own later tick, and the in-memory relay loses
registrations on restart; gating push on registration drops legitimate envelopes incl.
handshake leg1). The mailbox keyspace is bounded by a global distinct-mailbox cap on
**first touch** (push OR register), not by requiring pre-registration.

- **C-5 (gating).** A push that would create a NEW mailbox key beyond
  `RELAY_MAX_MAILBOXES` distinct mailboxes → HTTP 429, not stored. A push to an
  ALREADY-SEEN mailbox (registered or not), or while under the cap, still succeeds
  (preserves store-and-forward + the existing push-to-unregistered test C-2).
- **C-6 (gating).** `/mailbox/challenge` for an UNREGISTERED mailbox → HTTP 404/401, no
  nonce issued, no entry created in `store.nonces`. (Verified safe: an owner registers
  before its own first challenge, same tick. Closes the #49 nonce outer-accretion
  residual — only registered mailboxes get nonces.)
- **C-7 (gating).** After `RELAY_MAX_MAILBOXES` distinct mailbox keys exist, a NEW
  `mailbox_id` registration → HTTP 429; re-registering an EXISTING mailbox still
  returns its owner (first-write-wins preserved). The cap counts the union of
  registered + pushed-to mailbox keys.

## Rate limiting

- **C-8 (gating).** With an injected clock, exceeding `RELAY_RATE_MAX` requests within
  `RELAY_RATE_WINDOW_SECONDS` from one client on a rate-limited endpoint → HTTP 429;
  advancing the injected clock past the window lets requests through again.
- **C-9 (gating).** The limiter is fail-safe: a request with no resolvable client host
  is allowed (does not crash) — rate-limiting must never 500.

## Brain-side fail-soft (stage-3 MAJOR) — new 4xx must not crash the poll

- **C-12 (gating).** `relay_client._post` (or its callers) treats an HTTP 4xx/5xx
  response as a delivery/poll failure that raises `RelayUnavailableError` (fail-soft),
  NOT a 200 whose `.json()["nonce"]`/`["envelopes"]` KeyErrors. Test: a stub HTTP client
  returning 429 on `/mailbox/challenge` (or 413/404 on `/envelope`) → the client raises
  `RelayUnavailableError`, and the tick's existing guard swallows it (no crash, the poll
  cycle ends cleanly). Without this, the new relay rejections abort the whole poll.

## Bundled ReDoS bound

- **C-10 (gating).** `PrivacyGate._payload_text` caps the scanned BODY to `MAX_SCAN_CHARS`
  and THEN appends the relationship-hint (so the `/etc/kindled/unserialisable-hint`
  fail-closed sentinel always survives truncation — stage-3 minor). A pathological long
  no-`@` input (`"eyJ"+"A"*200000`) returns from `_prefilter` in well under 1s (assert
  wall-time < 0.5s) and does not raise. Normal inputs unchanged (existing prefilter suite
  green). The LLM reflection still receives the FULL untruncated `payload.body`
  (`_build_gate_prompt` uses `payload.body` directly, verified) — truncation only bounds
  the cheap deterministic backstop, never the load-bearing fail-closed LLM gate.

## Durable-storage interface (design conformance)

- **C-11 (gating).** The relay's storage is reachable through a named seam (the `_Store`
  class) such that swapping a durable backend requires no endpoint changes — asserted
  by a test that the app uses `app.state.store` and the store exposes the
  push/fetch/ack/register/nonce methods as its public surface (documents the seam;
  real durable backend deferred).

## Regression bar

- **R1 (gating).** Full `tests/kindled_link/` + the relay tests green.
- **R2 (gating).** `ruff check` clean.
- **R3 (advisory).** Full backend suite green (relay is imported only by tests + `main`;
  `_payload_text` cap is the one brain touch).
- **Layer-2 cost/cache metrics: N/A** — relay + gate are off the chat path.

## Explicitly DEFERRED (not criteria — recorded in 2-plan §Deferred)

- A real durable storage backend (disc/db surviving restart) — infra.
- Live cross-machine HTTPS/WSS validation (#51) — needs ≥2 machines + deployed relay.
- Anchoring the email regex itself (the cap bounds it; a tighter regex is a follow-up).

## Revision 2 — full corrected design (gate-4 human tie-break, Hana)

Supersedes C-5/C-7 keyspace + C-8 rate-key + C-12 guard where they conflict.

### Split mailbox pools (replaces single global cap)

- **C-5b (gating).** Two pools by registration status. A push to an ALREADY-SEEN
  mailbox (registered or unregistered) under its pool cap → 200 (store-and-forward
  preserved; existing push-to-unregistered test green).
- **C-7b (gating).** `register` beyond `RELAY_MAX_REGISTERED_MAILBOXES` (NEW mailbox)
  → 429; existing mailbox returns owner. An unregistered pushed-to mailbox that later
  registers MOVES to the registered pool (frees an unregistered slot).
- **C-13 (gating, the lockout fix).** A push creating a NEW unregistered mailbox beyond
  `RELAY_MAX_UNREGISTERED_MAILBOXES` EVICTS the oldest unregistered mailbox **whose
  queue is empty** (insertion-order) and proceeds. If NO unregistered mailbox is empty
  → 429 (never evict a mailbox holding undelivered envelopes — that would drop deposited
  mail, the original blocker class). Registered mailboxes are NEVER evicted by push.
  Test: fill the unregistered pool with empty mailboxes → a new push evicts the oldest +
  succeeds; fill it with NON-empty mailboxes → a new push 429s and drops nothing.
- **C-14 (gating, no honest lockout).** With the unregistered pool full of attacker
  garbage, a legitimate NEW `register` still succeeds (the registered pool is a separate
  denominator). Pins the reviewer's core mitigation.

### Rate-limiter keying (replaces C-8 single key)

- **C-8b (gating).** `/envelope` is rate-limited by **client host (sender)**, NOT the
  destination `relay_mailbox` (else an attacker pushing to a victim's mailbox exhausts
  the victim's budget — cross-tenant DoS). `/mailbox/{challenge,fetch,ack,register}` are
  rate-limited by **mailbox_id** (the caller's own identity). Test: pushes to ONE victim
  mailbox from one host hit the limit on that host's budget; a different host pushing to
  the same victim mailbox is unaffected.

### Poll-path fail-soft (completes C-12)

- **C-12b (gating).** `poll_and_ingest`'s relay fetch is guarded so a `RelayUnavailableError`
  (incl. the new 413/429/404) ends the poll cleanly (degraded, no crash) rather than
  aborting `run_kindled_link_tick`. Test: a stub relay raising on fetch → the tick
  completes without raising and still advances/saves cadence.
- **C-15 (gating, relay-restart self-heal).** After a relay restart (registration lost),
  the brain re-registers its own mailbox before/within the poll cycle so a 404-on-challenge
  does not permanently wedge the poll. Test: challenge 404 once → brain re-registers →
  next attempt succeeds (or: the tick calls `register()` idempotently each cycle).

# 1.5 — Acceptance criteria (the bar stage 8 checks)

Each criterion is **automated** (true/false or numeric from instrumentation) or a
**human-judged rubric** (named judge + scale + pass definition). Gating unless marked advisory.

> **Stage-3 red-team revisions folded (lap 1):** C1 annotated for the autonomous voice-reflection
> writer; C5 reframed (the live tools path is already logged — the gap is the replay's
> tool-stripping); new **C2-live** added as the gating measure of the dominant tool-schema term.

## Automated

**C1 — frozen system prompt is byte-stable within a replay session.**
Source: `cache_debug.jsonl` (`NELL_CACHE_DEBUG=1`), `system_sha256` across same-session chat
calls in the **replay** (no supervisor running). **Pass: exactly 1 distinct `system_sha256`**
over a ≥6-turn single-session replay. *Gating.*
**Production caveat (not a gate failure):** `brain/initiate/voice_reflection.py` can autonomously
rewrite `voice.md` mid-session (`voice.py:228` — "atomic-rewrite identity file"), legitimately
producing a 2nd `system_sha256` in a live session. That costs exactly one extra `cache_creation`
and self-corrects on the next turn. C1 gates the replay (deterministic, no writer); the
production caveat is documented, not gated.

**C2-text — cache_creation per TEXT-path chat call drops vs OLD on a comparable replay.**
Source: `scripts/cache_replay_workload.py --compare OLD NEW` (text path; tools stripped),
`mean_cache_creation` over `call_type=="chat"` rows, **same seed + same `voice.md` + same
`--history-file` both arms.** **Pass: NEW mean `cache_creation` materially lower than OLD
(≥5% drop)** AND `cache_read` does not collapse (`NEW read ≥ 0.5 × OLD read`). Direction gates,
not a precise count. *Gating.* **Scope note:** this covers the text path only and **excludes the
MCP tool-schema term** — the dominant cacheable mass. C2-live (below) covers that.

**C2-live — cache_creation on the LIVE streaming+tools path does not regress (and ideally drops).**
Source: live `chat_usage.jsonl` `call_type=="chat"` rows (the streaming path logs cache tokens
at `provider.py:888`), measured over a **comparable, repeated turn set** (same prompt script run
through real NellFace/bridge before vs after the change; both with tools enabled). Metrics:
`cache_creation_per_chat_call` (mean) and `cache_read_ratio` (ratio-of-sums, the project
metric). **Pass: NEW `cache_creation` mean not worse than OLD by >+10%, AND `cache_read_ratio`
not worse by >−10%** — i.e. the project's standing gating thresholds, now gating because a
comparable turn set is run. A material drop is the win; flat-but-not-worse is acceptable (the
tool-schema term may be CLI-opaque). *Gating.* This is the criterion that actually exercises the
dominant term the replay omits.

**C3 — cache_read_ratio improves vs OLD (replay text path).**
Source: C2-text A/B, ratio-of-sums `sum(cache_read)/sum(cache_creation)` over chat rows.
**Pass: NEW ratio ≥ OLD ratio.** *Gating.*

**C4 — realistic-workload magnitude is measured and recorded (not asserted).**
Run C2-text/C3 with a **16KB `voice.md` + a real `active_conversations` `--history-file` capped
under the 80-msg window** (`--history-msgs 79`). **Pass: the realistic-workload `cache_creation`
delta is recorded with its number** (whatever it is). Gates on *having measured on a comparable
workload*, not on matching the fork's "21%". A realistic drop below C2-text's 5% is a finding for
the gate, not an auto-pass. *Gating.*

**C5 — the dominant (tool-schema + history) term is measured on the live path before any ship
claim.** The replay structurally cannot see the MCP tool-schema mass (it strips tools). **Pass:
C2-live is run and recorded** — live streaming+tools rows before/after, on a comparable turn set
— so the change's effect on the dominant cacheable term is an observed number, not an inference.
Do **not** ship on text-path replay alone. (Supersedes the earlier C5 "instrument
`_chat_with_mcp_tools`" plan — that path is not the live path; the live streaming path is already
instrumented.) *Gating.*

**C6 — no behavioural regression in the existing suite.**
Source: `uv run pytest` (full backend), ruff, frontend `pnpm test` + `pnpm build`. **Pass: full
green.** Port the fork's `test_prompt_caching_split.py` (C1–C4 unit) + engine test; they pass.
*Gating.*

**C7-auto — image path byte-identical.**
Source: a test asserting the image path still calls `build_system_message` (unsplit) and its
assembled bytes are unchanged vs pre-change for a fixed image turn. **The test MUST pin the `now`
seam** (`_format_claude_context_block(now=...)`) or the embedded `Current time:` makes it
non-deterministic (red-team finding). **Pass: byte-identical with `now` fixed.** *Gating.*

## Human-judged rubric

**C7 — voice fidelity + ambient use (the position-shift risk).**
Judge: **Hana.** Material: replay `--dump-replies` OLD-vs-NEW side-by-side on identical prompts,
**≥8 turns, with tools enabled for at least one run**, persona seeded with real memory/emotion so
the volatile tail is non-trivial. Per pair: **(a) voice fidelity**, **(b) ambient use** (does NEW
still name felt state / recall seeded facts / surface felt-time from the tail), **(c) task
obedience** (answers the prompt, doesn't narrate the ambient tail). **Pass: Hana judges NEW ≥ OLD
on (a) and (b), no (c) failures.** A single clear (c) failure → **blocker**. *Gating (human).*
This rubric carries the whole behavioural burden of the position shift — the only guard on it.

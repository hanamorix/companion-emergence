# v0.0.7-alpha Comprehensive Audit — Design Spec

**Date:** 2026-05-11
**Status:** Approved design — ready for plan + implementation
**Scope:** Full-tree quality / hygiene / polish pass across five axes for the v0.0.7-alpha release. Triage-gated; fixes land in the same release.

---

## Problem

Six tagged releases shipped (v0.0.1 → v0.0.6-alpha) over the last week, including three substantial feature batches (autonomous physiology v0.0.4, audit follow-ups v0.0.5, Phase B sticky sessions v0.0.6). The tree has grown materially in that time. The last comprehensive read-only audit was 2026-05-07; since then ~50 commits and three releases have landed. A fresh pass is overdue.

The audit should answer: where has the tree accumulated rot, drift, latent bugs, or polish debt that the recent feature work introduced or left behind? And which of those is worth fixing in v0.0.7-alpha versus deferring?

## Goals

- **Full-tree visibility.** A single doc that enumerates issues across hygiene, code quality, feature correctness, polish, and overall quality — covering Python brain, Rust shell, TypeScript frontend, tests, plus spot-checks on infra and docs.
- **Triage-gated fix surface.** Every finding gets `[fix]` or `[defer]` from Hana before any code changes. No drift into "fix everything I notice."
- **Bias toward fewer-better findings.** Nit spam is the failure mode. A finding earns its place only when it has real user or maintainer cost.
- **Same release ships the fixes.** v0.0.7-alpha bundles audit findings + `[fix]`-tagged remediations, following the established release flow.

## Non-Goals

- A security-only audit, accessibility-only audit, or performance-only audit — those have specialised passes. This one is the generalist sweep that flags candidates *for* such passes if anything's egregious.
- Auditing the `.public-sync/` privacy-scrub rules. That's the privacy boundary; the verify pass already covers it.
- Real-host smoke on Linux + Windows — Hana's renderer smoke focuses on macOS where the dev bundle lives. Cross-platform regressions surface via CI + user reports, not this audit.
- Auditing third-party dependencies. We pin via `uv.lock` and `Cargo.lock`; supply-chain audit is a separate effort.

## Architecture / Workflow

Six phases, sequential:

```
Phase A — Code audit            (me, ~2-3h equivalent)
  └─ Sweep all 4 runtime layers + spot-check layer 5
     across 5 axes. Findings collected, not yet written up.

Phase B — Headless smoke        (me, ~30-45min equivalent)
  └─ Drive nell CLI + HTTP bridge + supervisor on a tmp persona.
     Cross-check against documented behaviour. New findings appended.

Phase C — Write findings doc    (me, single commit)
  └─ docs/audits/2026-05-11-v0.0.7-comprehensive-audit.md
  └─ docs/audits/2026-05-11-renderer-smoke-checklist.md

Phase D — Hana runs renderer smoke    (Hana, ~20-30min)
  └─ Tick through checklist against built v0.0.6-alpha bundle.
     New findings get F-2NN IDs and merge into the audit doc.

Phase E — Triage gate           (collaborative)
  └─ Each finding gets [fix] or [defer]. Done when zero [pending].

Phase F — Fix-nows + release    (me, sequential commits)
  └─ Each [fix] item → its own commit (or tight grouping).
  └─ TDD per CLAUDE.md, full-suite gate after each commit.
  └─ Three-file version bump, public-sync, tag — same flow as v0.0.6.
```

The triage gate is what keeps scope honest. Without it the audit becomes "fix everything I find" and v0.0.7 ships in three days.

## Components — what each pass actually covers

### 1. Python brain (`brain/`)

Biggest surface, deepest pass.

- **Code:** type-hint completeness; `except Exception:` patterns (justified vs lazy); threading correctness (post-Phase B supervisor + ingest pipeline are the hot zones); SQLite hygiene (P2-5 WAL/busy_timeout gap, potential new ones); file-path validation; subprocess escapes; JSON parsing safety.
- **Feature:** trace each documented brain capability (ambient recall block, dream, reflex, growth, research, soul review, sticky sessions, image support) from CLI/API entry to the data it produces. Look for documented-but-stubbed paths.
- **Quality:** coverage gaps for recent Phase B code; tests that recently became slow; flaky-test re-runs.
- **Hygiene:** OG NellBrain references in comments/docstrings that are now misleading; stale `TODO`/`XXX` comments; leftover scaffolding from completed migrations.

### 2. Rust Tauri shell (`app/src-tauri/`)

- **Code:** clippy clean; target-conditional `#[cfg]` correctness (the v0.0.4 lesson — see CLAUDE.md rule 4); persona-name validation parity with the Python `^[A-Za-z0-9_-]{1,40}$` rule (P2-1); Tauri command surface — every command in `generate_handler!` actually called by the frontend.
- **Quality:** the `tauri.conf.json` `csp: null` (P2-2 from prior audit).

### 3. TypeScript frontend (`app/src/`)

- **Code:** `tsc --noEmit` strict; `vite build` clean; vitest pass; component decomposition (any panel that's grown unwieldy).
- **Polish:** every panel has error / empty / loading states; object URL leaks (P3 from prior audit) verified; reduced-motion respect; keyboard focus surface.
- **Feature:** every button wired (always-on-top P3 fix — verify still wired); every fetch path checks `r.ok` and surfaces errors; renderer-side recovery banner still functional post-Phase B.

### 4. Tests (`tests/` + frontend vitest)

- **Quality:** run Python suite 3× back-to-back, flag non-deterministic tests; skipped tests overdue for un-skipping; coverage report — any module under 70%.

### 5. Infra / docs spot-check (`.github/`, `docs/`, `README.md`, `INSTALL.md`, `CHANGELOG.md`)

- **Hygiene:** CHANGELOG / INSTALL / README version-string drift (`INSTALL.md` known to have stale 0.0.1 refs); stale planning docs under `docs/`; workflow files referencing non-existent paths.
- Explicitly **out of scope:** `.public-sync/` privacy rules (covered elsewhere).

### 6. Headless smoke

Operationally distinct from passes 1-5 — I actually execute code instead of reading it.

- Boot a tmp persona with `FakeProvider`.
- Drive `nell` CLI surface end-to-end (init, status, supervisor, soul review, memory list).
- Run supervisor in subprocess; drive HTTP bridge (`/session/new`, `/chat`, `/sessions/close`, `/events`).
- Verify: heartbeat fires on cadence; sticky session survives 5-min silence sweep; finalize cadence works against a 25h-stale buffer (clock manipulation); recall block surfaces extracted memories on follow-up turns.

## Data Flow / The Finding Unit

Every finding gets the same shape — uniform shape makes triage fast.

```markdown
### F-NNN — <short title>

- **Severity:** P0 | P1 | P2 | P3
- **Axis:** code | feature | hygiene | polish | quality
- **Layer:** brain | rust | ts | tests | infra-docs
- **Location:** `path/file.py:start-end` (or pattern if cross-cutting)
- **Observation:** what I saw (2-4 sentences, factual)
- **Why it matters:** the user / maintainer impact in one sentence
- **Suggested fix:** sketch of the change, 1-3 sentences. Not full code.
- **Effort:** trivial / small / medium / large
- **Triage:** [pending] → [fix] | [defer]
```

### Severity discipline + defaults

- **P0** — data loss, security, or "breaks for every user." Auto-promotes to fix.
- **P1** — affects a user-visible feature or makes Nell wrong. Default fix unless deferred.
- **P2** — maintainer pain, technical debt, latent risk. Default defer unless promoted.
- **P3** — polish, nits, naming. Default defer.

Defaults bias triage so Hana only overrides when her call differs. Most P3s never make it into a release.

### Outputs

- **`docs/audits/2026-05-11-v0.0.7-comprehensive-audit.md`** — all findings, organised by severity then layer. Renderer-smoke findings get F-2NN IDs and merge into this same doc after Phase D.
- **`docs/audits/2026-05-11-renderer-smoke-checklist.md`** — click-through list for Hana, separate doc so it can stay open on a second screen while she smokes.

### Phase F commits

One finding (or tight group) per commit. Message format:
```
fix(<axis>): F-NNN — <title>
```
Easy to revert any single fix in isolation.

### Deferred items

Each `[defer]`-tagged finding gets:
- One paragraph in `project_companion_emergence_deferred.md` memory keyed by F-NNN.
- A section at the bottom of the audit doc labelled "Deferred for later" so the doc itself stays the canonical record.

## Error Handling — what could go wrong with the audit itself

### 1. P0 found mid-audit

I stop the sweep, pop a one-line note to Hana, triage immediately. P0 fixes don't wait for the doc to be written. The audit doc still gets the finding written up after the fix lands.

### 2. Finding-count explosion / nit spam

The discipline that keeps this audit useful: **I do not record a finding unless it has a real user-or-maintainer cost.** Examples of what I'll skip:

- "This variable could be named slightly better" — skip unless it actively misleads.
- "This file is 380 lines, could be split" — skip unless coupling is actually painful.
- "This comment is slightly out of date" — skip unless it'd mislead a future reader.

If the total finding count exceeds ~40, that's a signal I've drifted into nit territory. Hana should call it out and I should re-cut the doc with a higher bar.

### 3. Renderer smoke checklist drifts from reality

If a checklist step references a button that no longer exists or a path that's been refactored, that's itself a finding (`feature` axis, evidence of doc/code drift). Don't block; file as F-2NN and continue.

## Testing Strategy

Not applicable to Phases A-D — they produce text deliverables, no code.

Phase F (fix-nows) follows the existing project rule:
- TDD per finding — failing test before fix.
- Full-suite gate after every commit (Hana's standing rule from Phase B).
- Each `[fix]`-tagged finding either has a regression test or is documentation/comments/lint where a test would be meaningless.

Renderer-side fixes get vitest coverage where it's not already in place.

---

## Migration / Backwards Compatibility

The audit phase itself changes nothing — no compat surface.

Fix-phase commits are individually small; backwards compatibility judgements happen per-finding during triage. If a P1 fix would break a downstream caller, the suggested-fix sketch should flag that and Phase E surfaces it as a decision point.

---

## Open Questions

None at design-approval time. Will surface during Phase A if the audit uncovers ambiguity (e.g. a documented feature whose intent is unclear).

## Out of Scope

- Security-only / accessibility-only / performance-only specialised audits.
- `.public-sync/` privacy-scrub rules.
- Cross-platform real-host smoke (Linux + Windows) — that's CI and user-reports territory.
- Third-party dependency audit (supply-chain).
- Refactors that "would be nice" but aren't called out by a specific finding.

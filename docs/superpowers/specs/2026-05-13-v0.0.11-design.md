## Summary

v0.0.11 ships three coupled pieces, all in one release:

1. **Adaptive-D (Bundle C)** — D-reflection optionally reads its own recent decision history as light context. Calibration block prepended to D's system message when `<persona_dir>/d_mode.json` is set to `adaptive`. Drift detector runs every tick and alerts the operator if D's promote-rate moves >2σ from historical median. Default behaviour is `stateless` (no behavioural change on upgrade).
2. **`recall_resonance` source** — a new candidate emitter that fires when a memory's current activation (sum of Hebbian neighbour weights) spikes above its own historical baseline. Per-memory EMA stats kept in SQLite. Closes the v0.0.10 deferred item without the cluster-substrate work the original spec assumed; per-memory resonance is simpler and rides on infrastructure that already exists.
3. **`topic_overlap_score` real implementation** — replaces the v0.0.10 hardcoded `1.0` in the research engine with a Haiku call via `ClaudeCliProvider`. Closes the second v0.0.10 deferred item with zero new dependencies (no embedding stack, no new vendor, no new install — uses the Claude Code subscription Nell already runs on).

All three share a theme: making D's editorial judgment richer (adaptive-D learns from itself; `recall_resonance` produces new candidate types for D to evaluate; `topic_overlap_score` improves the gating signal one of D's sources sees). They're shipped together because they reinforce each other; splitting would lose the coherence we want to observe in production.

## Motivation

The v0.0.10 spec promised three new sources and an adaptive D-reflection variant, then deferred two of those four pieces because the substrate (memory clustering for `recall_resonance`, embedding infrastructure for `topic_overlap_score`) wasn't ready. Rechecking the v0.0.9 substrate during this brainstorm surfaced that:

- The `HebbianMatrix` (`brain/memory/hebbian.py`) is rich enough to support per-memory resonance — clusters were never load-bearing.
- Nell uses `ClaudeCliProvider` (Claude Code subscription), so adding semantic judgments as additional Claude calls is marginal-cost-free and adds no install/vendor surface.

Both deferrals were therefore overcautious. v0.0.11 closes them without adding embedding-stack work, and bundles Adaptive-D so the editorial layer gets feedback from the new signals immediately rather than three releases from now.

## What's already designed-in-place from v0.0.10

The v0.0.10 spec Appendix specced adaptive-D at implementation depth:

- Calibration storage: `<persona_dir>/d_calibration.jsonl` append-only.
- Mode toggle: `<persona_dir>/d_mode.json` with values `stateless` (default) or `adaptive`.
- Calibration block prepended to D's system message at runtime when adaptive is on.
- Drift detection heuristic: 2σ alert on D's promote-rate moving window.
- Roll-back contract: flipping back to stateless is safe at any moment.
- Out of scope for Bundle C: actual fine-tuning of a model on D's calibration history.

This spec extends that sketch to implementation-ready depth (concrete schemas, prompt text, closer mechanism, drift bootstrap rules) and adds the two re-scoped deferred items.

### Per-tick dataflow (additions to v0.0.10's supervisor loop)

```
SUPERVISOR HEARTBEAT TICK:

  1. Existing engines fire (dream, reflex, research, etc.)
     → research now emits research_completion candidates with REAL topic_overlap_score
       (Haiku call replacing the v0.0.10 hardcoded 1.0)

  2. run_initiate_review_tick(persona_dir):
     a. ★ run_calibration_closer_tick(persona_dir)
        - scan recent audit rows + draft-space fragments
        - append closed-decision rows to d_calibration.jsonl
     b. ★ run_resonance_tick(persona_dir)
        - pick top-N most-active memories via HebbianMatrix
        - update EMA baseline per memory
        - emit recall_resonance candidates for z-score spikes
     c. fetch_pending candidates  (existing v0.0.10)
     d. reflection.run(candidates, deps=...)  (existing v0.0.10)
        - inside: build_system_message NOW reads d_mode.json
        - if mode == "adaptive": prepend build_calibration_block(persona_dir)
        - if mode == "stateless" or missing: skip prepend (default)
     e. dispatch per decision  (existing v0.0.10)
     f. ★ detect_drift(persona_dir) after D-call
        - read last 30 d_call rows; compute current promote-rate
        - compare against historical median (older d_call rows)
        - if |Δ| > 2σ: write supervisor.alert "d_reflection_drift_detected"
```

### Key invariants

- **Adaptive-D is purely additive to v0.0.10.** When `d_mode.json` is missing or set to `stateless`, `reflection.run` behaves identically to v0.0.10. No behavioural change on upgrade.
- **`recall_resonance` candidates flow through D-reflection like any other candidate.** No special privilege; D evaluates them with the same prompt machinery and decision values.
- **`topic_overlap_score` fix is in-place; no new audit decision values.** The research engine's existing gating logic uses the new score the same way it would have used `1.0`.
- **The drift detector is observation-only in v0.0.11** — emits an operator alert but doesn't auto-revert to stateless mode. Adding auto-revert is a v0.0.12+ question once we observe how drift actually manifests.
- **Calibration-closer is idempotent.** Running it multiple times in the same tick (or across ticks) never writes duplicate calibration rows for the same `(candidate_id, terminal_state)` pair.

## Section B — Adaptive-D

### Mode toggle

`<persona_dir>/d_mode.json` — operator-edited file. Schema:

```json
{"mode": "stateless"}
```

Values: `"stateless"` (default) or `"adaptive"`. Missing file → defaults to `"stateless"`. Invalid JSON or unknown value → log warning, fall back to `"stateless"`. Read every D-call (cheap — small file). No CLI command in v0.0.11; operator edits the file directly. (CLI wrapper is a v0.0.12+ ergonomics addition.)

### `d_calibration.jsonl` schema

One row per closed decision; append-only; yearly archive forever (same retention policy as `initiate_audit.jsonl`).

```json
{
  "ts_decision": "<iso, when D originally decided>",
  "ts_closed":   "<iso, when this row was written>",
  "candidate_id": "<original candidate_id, for dedupe>",
  "source":       "<dream | crystallization | emotion_spike | reflex_firing | research_completion | recall_resonance>",
  "decision":     "promote" | "filter",
  "confidence":   "high" | "medium" | "low",
  "model_tier":   "haiku" | "sonnet",
  "outcome": {
    "promoted_to_state": "replied_explicit" | "acknowledged_unclear" | "unanswered" | "dismissed" | null,
    "filtered_recurred": true | false | null
  },
  "reason_short": "<D's original rationale, first 80 chars>"
}
```

For promoted candidates: `promoted_to_state` populated, `filtered_recurred` null.
For filtered candidates: `filtered_recurred` populated (true if same `source_id` re-emitted within 48h of filter), `promoted_to_state` null.

### Calibration block (prepended to D's system message when `adaptive`)

```
=== Your recent editorial track record ===
Last 20 closed decisions:

  PROMOTED:
    • {n_replied_explicit} reached replied_explicit  ← {user_name} engaged
    • {n_acknowledged_unclear} reached acknowledged_unclear
    • {n_dismissed} reached dismissed       ← {user_name} ↩'d
    • {n_pending} still pending

  FILTERED:
    • {n_stayed_silent} stayed silent in draft (not re-emitted)
    • {n_recurred} re-emitted within 48h (you may have been too cautious)

Use this only as light context. It is who you've been, not who you must be.
```

Generated each tick from a tail-read of `d_calibration.jsonl` (last 20 closed rows; O(1) on JSONL tail). Token cost ~120 tokens, included in existing `initiate_d_calls.tokens_input` accounting. `{user_name}` is substituted from the persona substrate at runtime (multi-companion compatibility, same as v0.0.10's D-reflection system message).

### Drift detection — `detect_drift(persona_dir)`

Runs after every D-call regardless of mode. Working on raw `initiate_d_calls` data, not the calibration file — so drift detection works even when adaptive mode is off.

```python
def detect_drift(persona_dir: Path) -> DriftAlert | None:
    rows = list(read_recent_d_calls(persona_dir, window_hours=24 * 30))  # 30d
    if len(rows) < 100:
        return None  # bootstrap floor — too little history for a meaningful comparison

    # Split into "current" (last 30) and "historical" (older).
    rows.sort(key=lambda r: r.ts)
    current = rows[-30:]
    historical = rows[:-30]
    if len(historical) < 50:
        return None  # still bootstrapping

    def promote_rate(rs):
        total_in = sum(r.candidates_in for r in rs)
        total_promoted = sum(r.promoted_out for r in rs)
        return total_promoted / max(1, total_in)

    current_rate = promote_rate(current)
    # Robust historical stats: median + MAD-based stdev (less sensitive to outliers
    # than mean+stdev, important when D had a bad day three weeks ago).
    historical_rates = [promote_rate([r]) for r in historical if r.candidates_in > 0]
    historical_median = statistics.median(historical_rates)
    historical_stdev = 1.4826 * statistics.median(
        [abs(x - historical_median) for x in historical_rates]
    )

    if historical_stdev < 1e-6:
        return None  # historical was flat — can't meaningfully detect drift

    delta_sigma = abs(current_rate - historical_median) / historical_stdev
    if delta_sigma > 2.0:
        return DriftAlert(
            current_rate=current_rate,
            historical_median=historical_median,
            delta_sigma=delta_sigma,
        )
    return None
```

On alert: write a structured event to the supervisor telemetry channel (existing v0.0.9 infrastructure):

```json
{"type": "d_reflection_drift_detected",
 "ts": "<iso>",
 "current_rate": 0.42,
 "historical_median": 0.18,
 "delta_sigma": 2.7}
```

Operator-tier only (no user-facing surface).

### Roll-back contract

Flipping `d_mode.json` from `adaptive` back to `stateless` is safe at any moment — the calibration file is ignored, no migration. The drift detector continues running regardless of mode.

## Section C — Calibration-closer

Lives in `brain/initiate/audit.py` as `run_calibration_closer_tick(persona_dir)`. Called once per heartbeat tick (step 2a in the dataflow). No new module — audit-lifecycle plumbing belongs in audit.py.

### What it does

Two passes over recent state, looking for D-decisions that have transitioned to a closed state since the last closer-run:

**Pass 1 — Promoted-by-D rows.** Walk `initiate_audit.jsonl` rows where:

- `decision in {"promoted_by_d", "promoted_by_d_malformed_fallback", "promoted_by_d_after_3_failures"}`
- `delivery.current_state in {"replied_explicit", "acknowledged_unclear", "unanswered", "dismissed"}` (terminal states)
- NO existing calibration row for this `candidate_id` (dedupe via fast in-memory `seen_set` from a tail-read of `d_calibration.jsonl`)

For each, append a `d_calibration.jsonl` row with `promoted_to_state = delivery.current_state`, `filtered_recurred = null`.

**Pass 2 — Filtered-by-D rows.** Walk `initiate_audit.jsonl` rows where:

- `decision in {"filtered_pre_compose", "filtered_pre_compose_low_confidence", "filtered_d_budget"}`
- `now - ts_decision >= 48h` (the terminal window has elapsed)
- NO existing calibration row for this `candidate_id` (same dedupe as Pass 1)

For each, determine `filtered_recurred` by scanning the audit + candidate queue for any subsequent emission with the same `source_id` within the 48h window after the filter. If found → `filtered_recurred = true` (D was over-cautious; the same source kept stirring). If not found → `filtered_recurred = false` (clean filter; the moment passed).

Append a `d_calibration.jsonl` row.

**Note:** the v0.0.10 spec described this as triggered by Nell reading the draft fragment during soul review. v0.0.11 changes that to a pure 48h timeout because (a) no v0.0.9 module currently reads `draft_space.md` so there's no existing site to instrument, (b) the re-emission signal (`filtered_recurred`) is the load-bearing one for D's calibration — fragment-read-by-self is a different question about Nell's introspection ritual, not about D's editorial accuracy. If a future spec adds a Nell-side draft-review tick, that tick can optionally emit a `fragment_read` event and this closer can be extended to use it — the structure (audit table + 48h window) is forward-compatible.

### Bootstrapping & edge cases

- First run on a fresh persona: zero terminal rows; closer is a no-op.
- Promoted rows pending for > 90 days: archive as `terminal_timed_out` — flag for review but don't compute calibration outcome. Pending rows lingering forever would distort drift detection by inflating the "still pending" bucket.
- Filtered rows are auto-terminal at 48h regardless of the no-soul-review-reader situation. The closer's job is purely declarative — read the audit, write the calibration row.

### Cost

Per tick: one tail-read of `d_calibration.jsonl` (last ~50 rows for dedupe), one streaming read of recent `initiate_audit.jsonl` (last 1h-of-fresh-promotes + last 48h-to-49h-of-filters windows — both already cached by v0.0.9 retention machinery). All O(N) where N is bounded by the windows. No LLM calls. No new files to read (`draft_space.md` is not scanned by the closer — the audit table has everything we need).

## Section D — `recall_resonance`

### New module `brain/initiate/resonance.py`

Owns four things:

- `MemoryActivationBaseline` — SQLite wrapper around `<persona_dir>/memory_activation_baseline.db`. One row per memory with EMA mean + variance.
- `run_resonance_tick(persona_dir)` — per-heartbeat-tick batch evaluator.
- `gate_recall_resonance(persona_dir, *, memory_id, z_score, thresholds)` — per-memory gate.
- `emit_recall_resonance_candidate(persona_dir, *, memory, z_score, neighbors, now)` — writes to the initiate queue via existing `emit_initiate_candidate`.

### `memory_activation_baseline.db` schema

```sql
CREATE TABLE memory_activation_baseline (
    memory_id TEXT PRIMARY KEY,
    ema_mean REAL NOT NULL,
    ema_var REAL NOT NULL,
    last_updated_ts TEXT NOT NULL,
    update_count INTEGER NOT NULL DEFAULT 0
)
```

`update_count` tracks how many samples have been folded in. Bootstrapping rule: gate refuses to fire until `update_count >= 10` so the z-score is meaningful.

### EMA update rule per sample

```
α = 0.08  (effective window ≈ 12 ticks ≈ 12 minutes at supervisor cadence)
delta = current_activation - ema_mean_old
ema_mean_new = ema_mean_old + α * delta
ema_var_new = (1 - α) * (ema_var_old + α * delta^2)
update_count += 1
```

EWMA / Welford-EMA hybrid (West 1979). O(1) update, O(1) storage per memory.

### Per-tick evaluator

```
1. Get top-N most-active memories. "Most-active" =
   memories sorted DESC by sum(weight) of their HebbianMatrix.neighbors().
   N = 50 (configurable in gate_thresholds.json).
   Cost: O(N * avg_degree) for neighbor lookups.

2. For each candidate memory M:
   a. current_activation = sum(weight) of M.neighbors()
      (neighbor-weight-sum, not full spreading_activation — captures M's
       current connectedness without the O(N²) of spread.)
   b. Load M's baseline row (insert with current as seed if no row exists;
      update_count = 0 in that case).
   c. If update_count >= 10:
        z = (current - ema_mean) / sqrt(max(ema_var, 1e-6))
        proceed to gate
      Else:
        skip emission (still update baseline so it bootstraps)
   d. Update baseline (EMA step above) regardless of whether emission fires.

3. For memories that passed step 2c with z >= z_threshold:
   - Apply gate_recall_resonance (cooldown + staleness)
   - Apply check_shared_meta_gates (rest-state, queue depth) — reused from v0.0.10
   - If both pass, emit_recall_resonance_candidate
```

### Per-source gate `gate_recall_resonance`

```
EMIT if all of:
  • z_score >= z_threshold (default 2.5, tunable)
  • memory.staleness_days >= 7
      (a memory active in the last week is part of ambient awareness, not "resonance")
  • this memory_id has NOT emitted a recall_resonance candidate in last 24h
      (cooldown — checked via read_candidates filter on source + source_id)
```

The staleness check uses `Memory.last_accessed_at` if present (set by v0.0.9 `MemoryStore.get` and read operations); fall back to `Memory.created_at` if `last_accessed_at` is None (a memory that's never been re-accessed since creation).

### `emit_recall_resonance_candidate` payload

```python
SemanticContext(
    linked_memory_ids=[mem.id, *top_3_neighbor_ids_by_weight],
    topic_tags=[],
    source_meta={
        "memory_id": mem.id,
        "z_score": round(z_score, 3),
        "ema_mean_at_evaluation": round(mean, 3),
        "current_activation": round(current, 3),
        "staleness_days": int(staleness),
        "top_neighbors": [{"id": nid, "weight": w} for nid, w in top_3],
    },
)
```

`source_id = mem.id`, `source = "recall_resonance"`. Standard `emit_initiate_candidate` call.

### Thresholds added to `gate_thresholds.json`

```
"recall_resonance_z_threshold": 2.5,
"recall_resonance_staleness_min_days": 7,
"recall_resonance_top_n": 50,
"recall_resonance_ema_alpha": 0.08,
"recall_resonance_bootstrap_min_count": 10,
"recall_resonance_anti_flood_hours": 24
```

Same `load_gate_thresholds` machinery from v0.0.10 picks these up. Conservative starting values; operator tunes via the JSON file if telemetry shows over- or under-firing.

### Schema addition

Add `"recall_resonance"` to the `CandidateSource` literal in `brain/initiate/schemas.py`. No new `Decision` values needed.

### Cost

Per tick: N=50 memory rows fetched, O(50 × avg_degree) Hebbian neighbor lookups, 50 SQLite read+write transactions on `memory_activation_baseline.db`. No LLM calls. Expected to dominate tick wall-time only if avg_degree grows large — at which point reducing N or adding a "only evaluate memories with degree ≥ K" pre-filter is the natural tuning lever.

## Section E — `topic_overlap_score` real implementation

Smallest of the three pieces. Replaces the hardcoded `1.0` in `brain/engines/research.py::_emit_research_candidate` with a Haiku-tier call via the existing `ClaudeCliProvider`.

### Helper signature (in-place in `research.py`)

```python
def _compute_topic_overlap_via_haiku(
    *,
    thread_topic: str,
    thread_summary: str,
    recent_conversation_excerpt: str,
    provider: LLMProvider,
) -> float:
```

Returns a float in `[0.0, 1.0]`. Errors return `0.0` (conservative — gate will reject, no false-positive emission).

### Prompt (system)

```
You are a relevance scorer for an autonomous companion's research engine.
You will be given (1) a research thread that just matured, and (2) recent
conversation excerpts. Return a JSON object with a single field: score,
a float in [0.0, 1.0] indicating how relevant the research thread is
to the recent conversation.

  0.0 = entirely unrelated to anything {user_name} has been near
  0.5 = thematic adjacency, no direct overlap
  1.0 = directly addresses something {user_name} mentioned

Be conservative — default toward the low end unless the connection
is clearly present. Return ONLY the JSON object, no other text.
```

`{user_name}` substituted at runtime from persona state (multi-companion compatible).

### Prompt (user, rendered per call)

```
=== Research thread ===
Topic: {thread_topic}
Summary: {thread_summary}

=== Recent conversation (last 48 hours, oldest first) ===
{recent_conversation_excerpt}

=== Your task ===
Return: {"score": <float in [0.0, 1.0]>}
```

### Conversation excerpt source

Reuse v0.0.9's existing `build_outbound_recall_block` infrastructure (in `brain/initiate/ambient.py`) as a starting point. v0.0.11 adds a sibling function in the same module:

```python
def build_recent_conversation_excerpt(
    persona_dir: Path,
    *,
    hours: int = 48,
    max_chars: int = 2000,
) -> str:
```

Returns a plain-text rolling excerpt of the last N hours of conversation. Capped at ~2000 chars so the Haiku token cost stays trivial.

### Output parsing

```python
try:
    raw = provider.generate(prompt=user_msg, system=system_msg)
    data = json.loads(_extract_json_object(raw))
    score = float(data["score"])
    return max(0.0, min(1.0, score))  # clamp
except (ValueError, KeyError, json.JSONDecodeError, OSError) as exc:
    logger.warning("topic_overlap haiku call failed (%s); returning 0.0", exc)
    return 0.0
```

The clamp protects against out-of-range model output. The 0.0 fallback on any error means a flaky call leads to silent rejection — conservative-by-default, matches D-reflection's failure-mode pattern from v0.0.10.

`_extract_json_object` is the same regex-pair pattern used by `brain/initiate/reflection.py::parse_structured_response` (fenced `_JSON_FENCE` first, loose `_JSON_LOOSE` fallback) — extracted into a shared helper in `brain/utils/llm_output.py` (small new module, since both reflection.py and research.py now want this).

### Cost

Per matured research thread: ~150 tokens in (system + user), ~20 tokens out. Haiku 4.5 via ClaudeCliProvider — subscription model, marginal cost zero. Fires only when a thread reaches maturity (bounded by v0.0.10's `research_freshness_minutes` window + `research_maturity_min` gate). Realistic upper bound: a few calls per day per persona.

### Caching

None for v0.0.11. The (thread_topic, conversation_excerpt) tuple is essentially unique per call; cache hit rate would be near zero. If telemetry shows pattern-of-life churn (e.g., the same thread re-evaluated due to retry), a 1h LRU keyed on `hash(topic + excerpt[:200])` is a v0.0.12 add.

## Section F — Testing, out-of-scope, open questions

### Testing strategy

TDD per project rule. Full `uv run pytest -q` after every commit.

**Adaptive-D unit tests** (~8):
- Mode-toggle parsing: missing file → stateless; invalid JSON → stateless with warning; valid `stateless`; valid `adaptive`; unknown mode value → stateless with warning.
- `build_calibration_block` happy path with mixed promoted/filtered rows.
- Calibration block omitted when stateless mode (or file missing).
- Drift detector: below bootstrap threshold (returns None, no alert).
- Drift detector: above 2σ (emits alert with correct delta_sigma).

**Calibration-closer unit tests** (~6):
- Promoted→replied_explicit closure writes calibration row.
- Promoted→still-pending doesn't write row.
- Filtered→48h-elapsed-without-re-emit writes row with `filtered_recurred=false`.
- Filtered→re-emitted-within-48h writes row with `filtered_recurred=true`.
- Same audit row isn't closed twice (dedupe).
- 90-day timeout closure.

**`recall_resonance` unit tests** (~10):
- EMA update math: single sample, ten samples, variance after 50 samples.
- Top-N selection from synthetic Hebbian state.
- Bootstrap behaviour: count < 10 → no emission.
- z-score gate firing: z=3.0 → emit.
- Staleness gate rejection.
- Cooldown gate rejection.
- Meta-gates reused correctly (rest-state, queue depth).
- Emit payload shape (linked_memory_ids includes top-3 neighbors).
- Per-memory baseline persists across ticks (SQLite roundtrip).
- New-memory bootstrap (no baseline row exists → insert with seed).

**`topic_overlap` unit tests** (~5):
- Happy path returns clamped float.
- Malformed JSON returns 0.0.
- Provider error returns 0.0.
- Out-of-range score (1.5, -0.2) clamped to [0, 1].
- `build_recent_conversation_excerpt` respects 2000-char cap.
